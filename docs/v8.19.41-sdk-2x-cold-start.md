# v8.19.41 — SDK 0.2.x cold-start tuning

## Symptom

After upgrading `claude-agent-sdk` to 0.2.87 (for Opus 4.8 effort tier
support — see v8.19.38), freshly-spawned worker daemons take noticeably
longer to produce their first turn. Observed cold-start window on dev
hardware: **5–10 minutes** between `agent_start` (daemon process up,
IRC client connected, channels joined) and the first `engaged` event
(worker SDK produced an AssistantMessage).

The daemon is otherwise healthy:
- Python process alive (`ps -p <pid>` shows the daemon).
- IRC connection established (NAMES query returns the worker as a
  channel member; the boss CLI `culture boss brief` confirms in-channel
  delivery).
- No exceptions in `~/.culture/logs/agent-<nick>.log`.

But:
- No `engaged` / `model_resolved` action in
  `~/.culture/daemon-log/<nick>.jsonl`.
- No file at `~/.culture/audit/<nick>.jsonl`.
- The watchdog (v8.18.4 / v8.19.25) fires `idle_warning` with
  `reason=never_briefed` after 90s of inactivity.

This is **not** a never-briefed case; the worker is genuinely warming
up.

## Likely cause

SDK 0.2.87 ships a substantially larger bundled CLI binary (203 MB vs
0.1.50's smaller binary) with new agentic features (adaptive thinking,
new tools, MCP improvements). The cold-start cost — bundled binary
init, MCP server registration, model handshake — appears to dominate
the first-turn latency. Once the SDK has produced its first
AssistantMessage, subsequent turns are normal speed.

## Mitigation

v8.19.41 extends `IDLE_GRACE_SECONDS` from `90` → `600` (10 minutes)
so the watchdog no longer fires a false `never_briefed` warning during
the cold-start window. Operators who want the old aggressive cadence
can override via env:

```bash
CULTURE_IDLE_GRACE_SECONDS=90 culture agent start <nick>
```

Workers DO eventually engage — the boss daemon spawned at 22:50:07
engaged at 22:58:29 (8m 22s after start) in the v8.19.41 test, with
the new SDK and no other changes. Two briefs sent before engagement
are queued by the daemon's IRC buffer; the worker processes them
as soon as the SDK is ready.

## Open questions for upstream

- Is the 5–10 min cold-start expected, or a regression in 0.2.87 that
  should be reported to Anthropic?
- Does the cold-start cost shrink with the SDK's lazy-init flags?
- Is there a warm-pool / pre-spawn pattern the daemon can adopt?

These remain open. The v8.19.41 mitigation is **defensive** — it
prevents the watchdog from creating noise during a working cold-start
window. The root cause (long warm-up) deserves an upstream
investigation in a future session.
